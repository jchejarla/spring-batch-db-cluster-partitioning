Distributed Spring Batch,
coordinated by your database

An open-source extension for scalable, fault-tolerant partitioned job execution across JVM nodes — using the relational database you already run. No Kafka, no RabbitMQ, no ZooKeeper.

Get started Read the JOSS paper View on GitHub

Why this extension

Standard Spring Batch remote partitioning needs a message broker and can't tell when a worker has silently died. This replaces that with the database — gaining visibility and recovery for free.

Decentralized master election

The node that launches a job becomes its master — no dedicated coordinator or single point of orchestration.

Proactive node awareness

The master queries the live worker count before partitioning, so work is split to match the cluster you actually have.

Database-driven coordination

All cluster state lives in standard SQL tables. Works with PostgreSQL, MySQL, Oracle, and H2.

Flexible partition strategies

Round-robin, fixed-node-count, or scale-up assignment — chosen to fit the workload at runtime.

Explicit state tracking

Every partition's lifecycle — pending, claimed, completed, failed — is recorded transactionally. No silently lost work.

Automatic failover

Missed heartbeats mark a node unreachable; its transferable partitions are reassigned to healthy workers.

How it works

Coordination flows through three tables that augment Spring Batch's existing schema.

1

Nodes register and heartbeat

Each instance records itself in BATCH_NODES on startup and refreshes a heartbeat timestamp on an interval.

2

The master partitions the work

It reads the active worker count, splits the step, and writes assignments to BATCH_PARTITIONS with a pending status.

3

Workers claim and execute

Each worker polls for its assigned partitions, claims them transactionally, runs the Spring Batch step, and updates status.

4

Failures are detected and recovered

If heartbeats stop, the node is marked unreachable and its incomplete, transferable partitions are reassigned automatically.

Install

Available on Maven Central. Add the dependency and configure your datasource — no changes to your existing job or step definitions.

<dependency>
    <groupId>io.github.jchejarla</groupId>
    <artifactId>spring-batch-db-cluster-core</artifactId>
    <version>2.0.1</version>
</dependency>

Using it in your job

Turn on clustering in configuration, then write a partitioner. Your existing steps stay exactly as they are.

1. Enable clustering in application.yml:

spring:
  batch:
    cluster:
      enabled: true
      node-id: ${HOSTNAME:batch-node-01}
      heartbeat-interval: 3000
      task-polling-interval: 1000
      unreachable-node-threshold: 15000
      node-cleanup-threshold: 60000

2. Define a partitioner by extending ClusterAwarePartitioner. You receive the live node count and pick a distribution strategy:

@Configuration
public class MyJobPartitioner extends ClusterAwarePartitioner {

    @Override
    public List<ExecutionContext> splitIntoChunksForDistribution(int availableNodeCount) {
        List<ExecutionContext> partitions = new ArrayList<>();
        for (int i = 0; i < 10; i++) {
            ExecutionContext ctx = new ExecutionContext();
            ctx.putLong("startRange", i * 100000L);
            ctx.putLong("endRange", (i + 1) * 100000L - 1);
            partitions.add(ctx);
        }
        return partitions;
    }

    @Override
    public PartitionTransferableProp arePartitionsTransferableWhenNodeFailed() {
        return PartitionTransferableProp.YES;
    }

    @Override
    public PartitionBuilder buildPartitionStrategy() {
        return PartitionBuilder.builder()
            .partitioningMode(PartitioningMode.ROUND_ROBIN)
            .build();
    }
}

3. Wire it into a partitioned step exactly as you would with standard Spring Batch — pass your partitioner to .partitioner(...), and the library handles distribution, claiming, and failover across the cluster. The full job configuration is in the README usage guide.

Runnable examples

The examples module ships two end-to-end demos you can run locally in minutes — the bundled H2 profile needs no external database or broker.

Range-sum job

Splits a numeric range into partitions, sums each on a different worker, and aggregates the result — the quickest way to watch distribution and failover happen.

ETL job

Reads rows from a CSV, distributes chunks across workers round-robin, and writes XML — a full chunk-oriented reader, processor, and writer pipeline.
